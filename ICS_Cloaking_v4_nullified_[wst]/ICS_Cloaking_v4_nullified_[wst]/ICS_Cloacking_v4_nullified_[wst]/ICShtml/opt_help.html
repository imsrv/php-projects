Cloaking Options Help

Cloaking Options:
The detection stages can be set to Active or Stopped in any combination you wish. A page request will pass through each stage that is set to Active to be checked against the criteria for that stage that may identify the request as being from a spider.

For example, if IP STAGE 1 is set to Active, since this stage checks only for an IP Address match, the page request will be checked against the first IP Address in the ICS database, if there is no match it will be passed on to the next Active stage. This process continues until one of the Active stages identifies the request. If all stages AND all /IP/=Directory=/UserAgent/& statements are exhausted without a match, then the human page is served.

  • There are two exceptions to this:
    • If Browser MultiStage Detection has been set to Active then a five stage check is done on the page request, if the footprint is that of a human visitor browser, then all further active stages are bypassed and the human page is served. This check goes far beyond just checking if it is a Mozilla agent, it can detect human visitors 99% of the time. The other 1% will pass through the active spider detection stages before being eliminated as a spider because it is a human visitor, and will be served the human page. Spiders will pass through Browser MultiStage Detection to the following active stages for detection, even if the spiders are using Mozilla as a user agent trying to emulate a regular visitor.

    • The other exception is Generic Spider Detection. If this stage is set to Active then the page request is checked for a generic spider match after all other active stages have been exhausted. This ensures that only spiders not matched in any prior stages are detected by this stage.
Browser MultiStage Detection:
Human Browser Footprint Detection:
Set this to Active to detect regular visitors through a mutlistage check. When the criteria is satisfied a human page is served otherwise it is passed on to the next active stage.

This stage can reliably detect human visitors and proxy servers which tend to carry a full visitor footprint, and since the spider detection process is avoided on a majority of hits by using this stage, it makes for a very responsive system. It is recommended that this stage be set to Active.
Use One Spider Directory:
Serve All Known Spiders From One Directory:
Set this to Active to serve pages to detected spiders from the same optimized pages directory. This option is useful if you do not want to serve content that is specific to each detected and identified spider, rather you would prefer to serve the same set of optimized pages to all detected spiders.

 From This Directory:
If Serve All Known Spiders From One Directory is set to Active, then a directory must be chosen to indicate where optimized pages will be retrieved from. The drop down box will display all directories that have been created in "SEdirect" for you to choose a directory to serve from.

If you choose not to serve all spiders the same optimized pages then set this to Match To Spider and the detected spider will be served from the directory identified in the statement with the matching IP address and User Agent.
IP MultiStage Detection:
IP STAGE 1:
Set this to Active to match spiders by IP address. If the IP address of the visitor matches one of the IP addresses or one of the IP address ranges in the ICS spider database, then a cloaked page will be served.

If you have set Serve All Known Spiders From One Directory to Active then the optimized page will be served from the directory specified in From This Directory. If Serve All Known Spiders From One Directory is set to Stopped, then the optimized page will be retrieved from the directory named on the statement with the matching IP address in the ICS database.

 IP STAGE 2:
Set this to Active to match spiders by reverse class "c" IP address and User Agent, both the reverse class "c" and the User Agent must be matched for an optimized page to be served.

The reverse class "c" match can detect a spider that comes from a range of 254 IP addresses as long as at least one IP address from that range is present in the ICS IP database and the User Agent defined with that IP address also matches.

If the reverse class "c" IP address of the visitor matches the class "c" of one of the IP addresses and the User Agent also matches on the same statement, then a cloaked page will be served.

If you have set Serve All Known Spiders From One Directory to Active then the optimized page will be served from the directory specified in From This Directory. If Serve All Known Spiders From One Directory is set to Stopped, then the optimized page will be retrieved from the directory named on the statement with the matching class "c" IP address and User Agent in the ICS database.

 IP STAGE 3:
Set this to Active to match spiders by reverse class "c" IP address, the reverse class "c" and all of the elements of a known spider footprint must exist for an optimized page to be served.

IP STAGE 3 can detect a spider when it carries an IP address from within a range of a known class "c" and it leaves a footprint with all of the elements of a known spider, at least one IP address from that range must be present in the ICS IP database for a match to occur.

If the reverse class "c" IP address of the visitor matches the class "c" of one of the IP addresses and all of the elements of a known spider footprint, then a cloaked page will be served.

If you have set Serve All Known Spiders From One Directory to Active then the optimized page will be served from the directory specified in From This Directory. If Serve All Known Spiders From One Directory is set to Stopped, then the optimized page will be retrieved from the directory named on the statement with the matching class "c" IP address in the ICS database.
User Agent MultiStage Detection:
UA STAGE 1:
Set this to Active to match spiders by User Agent and full spider footprint. If the User Agent of the visitor matches a User Agent in the ICS spider database and a full spider footprint exists then a cloaked page will be served.

Note that this stage is not plain old User Agent cloaking. It cannot be spoofed by someone simply changing their browser's User Agent to match that of a known spider.

If you have set Serve All Known Spiders From One Directory to Active then the optimized page will be served from the directory specified in From This Directory. If Serve All Known Spiders From One Directory is set to Stopped, then the optimized page will be retrieved from the directory named on the statement with the matching User Agent in the ICS database.

 UA STAGE 2:
Set this to Active to match spiders by User Agent and partial spider footprint. If the User Agent of the visitor matches a User Agent in the ICS spider database and a partial spider footprint exists then a cloaked page will be served.

Note that this stage also is not plain old User Agent cloaking. It cannot be spoofed by someone simply changing their browser's User Agent to match that of a known spider. With this stage it is possible for a User Agent spoofer to have the page revealed, but the elements required to be present in the header of the request are not set by any spoofers that have been captured to date.

If you have set Serve All Known Spiders From One Directory to Active then the optimized page will be served from the directory specified in From This Directory. If Serve All Known Spiders From One Directory is set to Stopped, then the optimized page will be retrieved from the directory named on the statement with the matching User Agent in the ICS database.

 UA STAGE 3:
Set this to Active to match spiders by User Agent alone. If the User Agent of the visitor matches a User Agent in the ICS spider database then a cloaked page will be served.

Note that this stage is plain old User Agent cloaking. It can be spoofed by someone simply by changing their browser's User Agent to match that of a known spider.

If you have set Serve All Known Spiders From One Directory to Active then the optimized page will be served from the directory specified in From This Directory. If Serve All Known Spiders From One Directory is set to Stopped, then the optimized page will be retrieved from the directory named on the statement with the matching User Agent in the ICS database.
Generic Spider Detection:
GE STAGE 1:
Set this to Active to match spiders by full spider footprint. If a full spider footprint exists then a cloaked page will be served.

Note that this stage does not use any of the IP addresses or User Agents in the ICS db. This stage cannot be spoofed by someone simply changing their browser's User Agent to match that of a known spider. However it is possible for a User Agent spoofer to have the page revealed, but the elements required to be present in the header of the request are not set by any spoofers that have been captured to date.

Since we have no idea which spider was matched in this stage, only that it is a spider, a directory must be specified to serve generically optimized pages from. This is described below.

 Generic Match Served From:
If GE STAGE 1 is set to Active, then a directory must be chosen to indicate where generically optimized pages will be retrieved from. The drop down box will display all directories that have been created in "SEdirect" for you to choose a directory to serve from.

If you choose not to activate GE STAGE 1 then set Generic Match Served From to Stage Disabled. It is recommended that this stage be initially set to Stopped, and that Log All Unknown Spider Footprints be set to Active so that you can check what spiders you would be serving a generically optimized page to. Please refer to the section below on Log All Unknown Spider Footprints for additional information.
Notification And Logging Options:
Email Notification Of Spider Visits:
Set this to Active to have ICS4.0 send a e-mail notification when a detected spider makes a request for one of your optimized pages. The e-mail will provide information on the spider making the request, and will identify which active stage detected the spider as well as which directory the page content was retrieved from. It is recommended that this stage be set to Active.

 New IP Address Detection:
Set this to Active to have ICS4.0 send a e-mail notification when a known spider makes a request for one of your optimized pages and it is using an IP address that is not in the ICS db you are currently using. The e-mail will provide information on the spider making the request as well as the new IP address. This email will be in addition to the one you will receive for the spidering of the page if Email Notification Of Spider Visits is set to Active. It is recommended that this stage be set to Active.

 Log All Unknown Spider Footprints:
Set this to Active to have ICS4.0 write to a log (footprints.log) visits from spiders not detected in any of the active stages. Please note that the spiders captured here carry a footprint that requires less stringent criteria to match than those captured by GE STAGE 1, therefore some of the spiders recorded in this log will never match any of the detection stages including GE STAGE 1 Unless you specifically add their IP address and/or User Agent to the ICS db. The spiders that would be matched by GE STAGE 1 are identified in the footprints.log with two asterisks "**" as a prefix and suffix to the User Agent and the IP address fields.

Of course if GE STAGE 1 has been set to Active then no entries in the footprints.log will have the asterisks because they will have been detected by that stage. Hits appearing in this log have been served a human visitor page. It is recommended that this log be set to Active.

 Log Known Spider Visits:
Set this to Active to have ICS4.0 write to a log (spider.log) whenever a spider detected by any one of the active stages requests a page. Hits appearing in this log have been served a cloaked page. Depending on your settings the page has been served either from a specific spider directory, from the same directory for all spiders, or if detected by GE STAGE 1 from the directory specified for generic spider pages. It is recommended that this log be set to Active.

 Log Human Visits:
Set this to Active to have ICS4.0 write to a log (visitor.log) whenever a visitor is detected by Human Browser Footprint Detection or the visitor is not matched by any of the subsequent Active stages. Hits appearing in this log have been served a human page. Details of the hit will be captured which include referrer and keywords used at search engines. It is recommended that this log be set to Active.
